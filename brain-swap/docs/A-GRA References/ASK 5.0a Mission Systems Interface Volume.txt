1 MISSION SYSTEMS INTERFACE
The Mission Systems (MS) interface is the connection between MA and onboard MS such as
capabilities (e.g. Skills), sensor subsystems, payloads, and Mission Data processing services. The
major capabilities of the current L1 MS interface include the following:
•

Data exchanges between A-GRA compliant, Mission Autonomy equipped platforms utilizing
onboard Mission Systems

•

Providing status data to MA such as position, navigation, and timing (PNT) data, and system
status of the local and remote platforms

•

Providing MS-generated operational data such as entities tracked by MS systems

The L1 MS Interface captures the UCI-based interactions that MA will use to control and interface
with the both the local Mission Systems and other MA equipped ACP's. The design intent is to
provide a succinct, service-based interface to capabilities provided by the MS that MA will interface
with. The message set and content is intended to capture the minimal information exchange between
MA and MS to support the capabilities required for the decomposed system use cases. Concerns
that can be managed solely by MA, such as which ACP is the leader of an ACP team, are not
exposed by MA in the L1 interface. Similarly, the L1 interface abstracts the detailed operation and
implementation of MS subsystems such as network interfaces, and details such as the exact route
messages take to reach the node which is the leader are not exposed by MS. The Mission Systems
interface relies on OMS but A-GRA does not make assumptions about whether the MA and MS are
within a single mission package using a single ASB, or they operate as two separate mission
packages connected by an isolator. The A-GRA also does not make any assumptions about the CAL
or ASB chosen by either.

1.1 Interface Description
Below we arrange the MS to MA interactions into a set of functional groups we refer to as services.
These groupings are intended to help describe the capabilities the MS must provide and how they are
accessed. They do not imply any requirements or restrictions on the internal architecture or
implementation of MS. For each category, message definitions and interactions are detailed in the
following section. We anticipate further refinement and extension will occur in future versions of the AGRA.
Communications Service
The A-GRA L1 interface to MS seeks to enable communication services while isolating MA from the
detailed management of the network, communication terminals and the like to the extent possible.
The L1 interface defines message exchange with network endpoints currently visible to MS via
publication/subscribe or a directed message. This is described below in Decentralized Message
Service. To support that functionality, the Communication Service interactions provide insight into the
network state. To connect to remote nodes on currently accessible network interfaces, MS may use a
network discovery protocol, a pre-loaded configuration file, or other mechanism. Once connected, MS
provides MA with a notification that a new node is accessible. MS provides an system identifier,
analogous to a hostname in the case of an IP network, by which MA can direct messaging to the
remote node. MS holds the detailed network and addressing information, making use of it when MA
makes a request involving a particular system identifier. Once messaging is established between

nodes, MA may exchange information with remote nodes about node type, capabilities, and other
information as defined by the A-GRA MA L1 interfaces (C2, Peer). Keeping the addressing
information opaque to MA in this way enables implementations of the Communications and
Decentralized Message interactions to provide service across a set of heterogeneous networks.
Methods that nodes use to exchange their system identifiers, or for consistent identifiers to be
assigned for nodes on TDL or other networks that may not have the capability to provide an system
identifier, has been considered and implemented. As that functionality is within the MS boundary, it is
not currently included in the A-GRA interface. The comms terminal will automatically configure based
on a predetermined plan. The A-GRA does not currently require that an MA implementation make any
use of the comms plan, though information such as any periods that lack comms may be of use. The
L1 MA-MS interface does include an optional set of messages by which MA may request information
on comms systems configuration and dynamically adjust that configuration. We do not anticipate use
of these messages for the current A-GRA mission use cases. Longer term, more complex resource
interactions between use of multi-function apertures for comms and sensing, or contention between
vehicle pose and comms terminal usage, may require more detailed interaction. We anticipate these
optional interfaces may require revision as those use cases are considered in more detail in the
future.
Decentralized Messaging Service
The L1 interface with MS defines interactions to send and receive messages between A-GRA
compliant systems currently reachable by MS, either directly or via multiple network hops. Both
publish/subscribe and directing a message to specific recipients are supported. The interface is
designed to be consistent with the message-passing paradigm of OMS while including additional
information necessary to operate on over-the-air networks. This includes message priority for
bandwidth management and queuing as well as optional subscription filtering to eliminate
unnecessary propagation of messages. The interface is designed to allow implementations to operate
across heterogeneous network types (e.g. IP and non-IP) by utilizing generic asset IDs that can be
mapped by the Decentralized Messaging Service onto multiple addressing schemes and by not
defining interactions that require a connection-based network. The MS messaging interface assumes
that the message supplied by MA for transport is a line encoded payload to be delivered to the
remote A-GRA compliant mission package. This simplifies the L1 interface and decouples the MS
and MA schemas should they differ. Integration with assets that are not A-GRA compliant, and a
linked send/respond message pair have been considered but are not included in the current ASK.
Status Service
MS provides messages by which MA can request receipt of health, status, comms, or built-in test
(BIT) information for MS subsystems. MS will also report health, status, and BIT information of its
subsystems to MA at a configurable periodic rate.
PNT Service
MS provides messages by which MA may request and receive platform position and current time. MA
can also adjust the frequency PNT data is published.
Data Fusion Service
MS provides messages by which MA may receive processed tracks that are compiled from multiple
sensor inputs.

1.2.4 Data Fusion Package
1.2.4.1 Distribute Single Sensor Track Data to Perform Local Fusion
This interaction describes how MS provides tracks from onboard sensors to MA to enable local
fusion. This is accomplished via the EntityMT message, which contains the kinematic data of the
track. Additionally, the ObservationMeasurementReportMT message contains the sensor data that is
used to generate the track or entity. Track data sources include both onboard cooperative sensors
(e.g., Mode 5 IFF, ADS-B) and onboard non-cooperative sensors. It is important to note that only a
subset of the total Observation Measurement Reports (OMRs) recorded are associated with a given
track, and can therefore be referred to as Associated Measurement Reports (AMRs). These
associated OMRs are identified in the EntityMT by the MeasurementID field Once fusion has taken
place, the resultant fused tracks can then be utilized by and integrated into the Common Operating
Picture (COP). Alternately, if a platform is incapable of performing local fusion, the pre-fused tracks
may still be used by COP
Additionally, a kinematic and Combat ID (CID) fusion implementation will likely want the capability
that OMRs were generated from, and which system/subsystem had the capability generating the
OMR
Within the ObservationMeasurementReportMT message, there are two optional paths to different
sets of SubsystemIDs and SystemIDs
1) ObservationMeasurementReportMDT → ObservationMeasurement → MeasurementSource →
ElementDetails → SourceElementIdentifier → SystemID + SubsystemID
• These messages provide the system/subsystem that's the source of the OMR's kinematics.
2) ObservationMeasurementReportMDT → ObservationMeasurement → MeasurementSource →
SourceIdentity → CapabilityID + SystemID + SubsystemID
• These messages provide the system/subsystem and capability for the source of identity in the
OMR (it's expected this will usually be the same system/subsystem as the kinematics). A client
would want to know the capability type of the provided entity ID, and may have to subscribe to
additional messages to understand the type (e.g. [Capability]StatusMT messages).

1.2.4.2 Receive Processed Track
The Receive Processed Track sequence diagram describes how MA receives track information from
an onboard data fusion service. The basic functionality of a data fusion service is to process and
merge target reports from various internal and external sources to provide a resulting track, called
Entity. Fields that assist MA on decision making include Source, EntityStatus, Identity, Kinematics,
and EstimatedKinematics. This information, processed from 2D unfused tracks or 3D tracks (cues)
via EntityMT, not only provides MA a picture of the battlespace, but also includes information about
an entity's kinematic characteristics to support a variety of mission objectives, such as following an
Entity. This sequence shows a loop of EntityMT messages as long as there is more than one Entity to
convey to MA. This message is also triggered when new tracks are determined, or there are changes
to already established tracks.
Losing a track is a special optional case that happens when a sensor can no longer detect and track
an entity; when this occurs, the EntityLostMT message will notify Mission Autonomy of this event.

1.2.5 Decentralized Messaging Service Package
The Decentralized Messaging Service (DMS) Interface is a construct that exists to mux data between
MA and the MS. This interface facilitates the passing of encoded data payloads to/from C2 and
between ACP's in accordance with over the air network protocols. In the case when a link-specific
encoding must be used, such as Link 16, that encoding will be specified separately from the general
offboard message format. The Minimum Message Set (MMS) accounted for in the DMS consists of
the messages required to transmit data payloads off-board (MA_TxDataPayloadCommandMT,

Interface

Mission Systems Interface Volume (v. 5.0a)
MA_TxDataPayloadCommandStatusMT), a message to pass a received message from MS to MA
(MA_RxDataPayload), a set of messages for MA to indicate desired subscriptions to MS
(MA_MessagingSubscriptionSettingCommandMT,
MA_MessagingSubscriptionSettingCommandStatusMT) and a message for MS to send existing
subscription information to MA (MA_MessagingSubscriptionStatusMT). MA will populate
MA_TxDataPayloadCommandMT with an encoded data payload and optionally a list of network
endpoints by SystemID to send to MS. MA will send this MA_TxDataPayloadCommandMT for all
messages that may be sent off-board via a subscription by an external system. In order to send the
message off-board, MS will use the DMS to send the message OTA to another system's DMS. The
receiving system's DMS is responsible for unpacking the message, formatting it as a
MA_RxDataPayloadMT message, and forwarding the message to the appropriate interface for
decoding and processing. In the event that the receiving system is an ACP, the DMS will forward the
MA_RxDataPayloadMT message to MA if the message was intended for this platform by subscription
or directly addressed by the sender MA. MA will then unpack and and decode the EncodedPayload
field to form the intended MA message type.
In order to successfully send data between instances of DMS via the OTA network, the Data
Distribution Service (DDS) standard shall be used as the middleware. DDS provides a scalable, realtime, and decentralized publish-subscribe communication model well-suited to dynamic and
distributed environments. It allows systems to exchange structured data via shared topics in a flexible
deployment, resilient to node failure, and provides easier integration.
All implementations of the DMS must conform to the DDS specification (see
https://www.omg.org/spec/DDS/1.4/) and be compatible with the Real-Time Publish-Subscribe(RTPS)
wire protocol, minimum version 2.3. While a specific DDS implementation is not mandated, all system
components must comply with the interface and behavioral expectations described herein to ensure
interoperability. This section covers discovery mechanisms, topic definitions, quality of service
requirements, serialization and data encoding, and security features.
To enable interoperability, it is necessary to standardize several aspects of the DDS configuration.
First, two types of DDS topics will be used:
1. Pub/sub topics shall be named according to the UCI message type that will be sent via those
topics. "SystemStatus" or "ActionCommand" are valid Topic names. "RandomTopicName" or
"SystemStatusMT" are not valid Topic names. Subscriptions to these topics can be managed via the
“Subscribe to Message Topic” interaction.
2. Specific destination topics shall be used to send messages according to
MA_TxDataPayloadCommandMT.SpecificDestinationID fields. Every DMS shall subscribe to a DDS
topic that is equal to the UUID of its own SystemID. Since DDS uses pub/sub rather than point-topoint semantics, these topics will be used to send OTA messages to systems listed in
SpecificDestination fields of the MA_TxDataPayloadCommand. These topics will use RELIABLE
QoS settings. Topic names that correspond to SystemIDs will be prefixed with "id_", use all
uppercase letters, and use "_" instead of "-" for separators to comply with DDS Topic naming rules.
A SystemID of "3b39Bb83ee-d994-5f64-ae73-e62df8ec3c72" becomes a topic name of
"id_B39B83EE_D994_5F64_AE73_E62DF8EC3C72".
DMS implementations shall perform content filtering to ensure that duplicate MA_RxDataPayloadMT

1.2.5.1 Receive Encoded Payload Message
When the Mission System receives an encoded message from C2 or a peer ACP it can send the
encoded message to Mission Autonomy using MA_RxDataPayloadMT. The MA_RxDataPayloadMT
contains a hexBinary field called EncodedPayload that allows for a payload that is decoded by
Mission Autonomy. This message also has message fields RxDataPayloadID,
DataPayloadOriginator, Priority, MessageType, DestinationRouting, SpecificDestinationID,
Timestamp and MessageExpiry. RxDataPayloadID is a unique ID for this received data payload
message. The DataPayloadOriginator field in the MA_RxDataPayloadMT specifies the ID of the ACP
that originally sent the encoded message. The Priority field indicates the priority of a received
message using a number as follows: 0-10 indicates high priority, 10-20 indicates medium priority, and
20-30 indicates low priority. MessageType specifies the UCI type of the encoded payload.
DestinationRouting documents whether the message was routed only to specific destinations or also
to the topic corresponding to the MessageType. SpecificDestinationID is optional, but one or more
SpecificDestinationIDs will be included if the message was send to those systems directly. The
Timestamp field indicates the time at which the encoded message was received and the
MessageExpiry field indicates when the encoded message expires. The fields contained within the
MA_RxDataPayloadMT enable implementers to receive encoded payloads for facilitating
communication with other ACPs.

1.2.5.3 Send Encoded Payload Message
When the Mission Autonomy (MA) needs to send or publish an encoded message to an external
system, such as a peer or group of peer ACPs, it can send the encoded message to Mission Systems
(MS) using a MA_TxDataPayloadCommandMT message. The MA_TxDataPayloadCommandMT
contains a hexBinary field called EncodedPayload that allows for a payload to be sent to another
ACP's MA to be decoded. The MA_TxDataPayloadCommandMT message has fields MessageType,
Priority, Timestamp, MessageExpiry, SpecificDestinationID, and DestinationRoutingfields. The
MessageType indicates the UCI type of the encoded payload. The Priority field indicates the priority
of a message being sent using a number as follows: 0-10 indicates high priority, 10-20 indicates
medium priority, and 20-30 indicates low priority The Timestamp field indicates the time at which the
encoded message was received and the MessageExpiry field indicates when the encoded message
expires. The SpecificDestinationID field is either a list of the ACP endpoints the data is to be sent to
or is an empty list. The DestinationRouting field specifies whether the message should be sent only to
SpecificDestinationID destinations or to both the appropriate topic (based on MessageType) as well
as the SpecificDestinationIDs. If DestinationRouting=SPECIFIC_DESTINATION_ONLY then at least
one of SpecificDestinationID must be provided. Otherwise, a failure status message will be sent to
MA. The fields contained within the MA_TxDataPayloadCommandMT enable implementers to send
encoded payloads for facilitating communication with other ACPs. One or more accompanying
MA_TxDataPayloadCommandStatusMT messages are returned from MS to MA to indicate the status
of the process.

1.2.5.5 Quality of Service
To meet system requirements for reliability, latency, and data freshness, each topic, on the DDS
network, is governed by a set of Quality of Service (QoS) policies. These policies control how data is
transmitted, stored, and delivered between systems. At a minimum, all topics must declare their QoS
settings explicitly.
The default QoS policies include:
• RELIABLE delivery for critical data. This includes messages that require a response, such as
acknowledgments or other types of responses, ensuring that messages are delivered and
acknowledged, providing guaranteed delivery. Specifically, this applies to UCI primitive types:
Command-2, ActionRequest-2, DataRequest-2, DataRecord-1, and the
MA_SystemNotificationMT, OperatorNotificationMT, and OperatorNotificationAckMT
messages. This ensures that messages are delivered and acknowledged, providing
guaranteed delivery.
•

BEST_EFFORT delivery where data loss is acceptable. This includes messages that do not
require an acknowledgment or response. Specifically, this applies to UCI primitive types: Data1 and Status-1.

QoS compatibility is essential for interoperability and successful communication, ensuring that a
subscriber's QoS expectations are compatible with those of the publisher.

1.2.5.6 Security
System communication requires authentication, encryption, and access control to meet security
requirements. The DDS Security specification (see https://www.omg.org/spec/DDS-SECURITY/1.2/)
shall be leveraged to enforce these measures. Systems must be authenticated using X.509
certificates or equivalent credentials before joining the network. Implementers are responsible for
managing keys, certificates, and revocation policies.
All data transmitted over the network must be encrypted to protect it from eavesdropping and
tampering. Implementers must use the DDS Security plugins to enforce Transport Layer Security
(TLS) with mutual TLS (mTLS). This ensures data confidentiality, integrity, and authenticity during
transmission.

1.2.5.7 Serialization and Data Encoding
Serialization and data encoding are critical for ensuring interoperability between different systems in
the DMS. The DDS middleware supports various serialization formats to encode data structures
defined in the Interface Definition Language (IDL). The Common Data Representation (CDR) format
shall be used for serialization due to its interoperability, performance, and compatibility with DDS. All
data structures shared between systems must be defined in the IDL and serialized according to the
defined schema.

1.2.5.8 System Discovery
DDS enables distributed components to automatically discover each other on the network and
exchange metadata about the topics they publish or subscribe to. The DDS middleware relies on the
RTPS (Real-Time Publish-Subscribe) protocol, specifically the Simple Participant Discovery Protocol
(SPDP) and Simple Endpoint Discovery Protocol (SEDP), to dynamically form a mesh of
communicators without requiring a centralized broker. Upon startup, each node registers itself and
advertises its capabilities using these discovery protocols.

1.2.5.9 Topic Publication and Subscription
Communication between system components occurs over well-defined topics. Each topic represents
a stream of typed data, with any number of publishers and subscribers. Topics are uniquely named
and associated with a specific data structure defined in an Interface Definition Language (IDL). All
systems must agree on this schema and the semantics of each topic they interact with.
As described previously, each message type is assigned a dedicated topic, with the topic name
exactly matching the message name. This enforces a consistent mapping between data and
transport, simplifies subscription management, and allows fine-grained QoS control. Publishers and
subscribers may join or leave at any time, and DDS guarantees delivery according to the configured
QoS policies. Additionally, each DMS subscribes to a topic that is equal to its host system’s
SystemID.UUID to facilitate SpecificDestinationID messaging.
All topics transmit data using the MA_DataPayloadWrapper IDL type. The actual message content is
serialized into its EncodedPayload field. The IDL of the MA_DataPayloadWrapper is included in the
Autonomy Starter Kit release with the Decentralized Messaging Service component of the A-GRI.
MA will send all MA_TxDataPayloadCommandMT messages to MS for any message that could be
subscribed to. The DMS will advertise topics on the DDS interface based on the Topic field set in the
MA_TxDataPayloadCommandMT, but the message format on that topic will be the
MA_DataPayloadWrapper IDL that is filled with the contents from that command message. The DMS
will handle subscriptions by publishing messages on the appropriate topics to go off-board. For
subscriptions with specific SystemIDs, the DMS will filter and forward the relevant
MA_RxDataPayloadMT messages to MA.

1.2.5.10 Transport Protocols
To ensure robust and flexible communication, systems must support both UDP and TCP transport
protocols. TCP is recommended for scenarios requiring reliable, ordered delivery of messages. It
ensures data integrity and is suitable for messages that require acknowledgments or responses. UDP
is recommended for scenarios where low latency and minimal transmission overhead are critical. It is
suitable for non-critical data where occasional message loss is acceptable and where speed is
prioritized over reliability.

1.2.8.7 Publish External Weather Reports
MS can request weather data from off-board, periodically-updated weather services through comms
channels (ex: XM weather from satellite). MA can initiate a request of that data by sending MS
DataUpdateRequestMT with the RequestType set to Query, Query.MessageType set to WEATHER,
and Query.Query included. The query will contain details for specifics of the off-board weather data
being requested.
MS will respond with DataUpdateRequestStatusMTs to confirm that it has received the
DataUpdateRequestMT from MA and to subsequently provide updates on the state of the request
using the RequestProcessingState field. Finally, once MS has received the updated weather data, it
will send it to MA as a WeatherMT message, and send a final DataUpdateRequestStatusMT with
RequestProcessingState set to COMPLETED. If MS was unable to obtain the updated weather data,
it will not send WeatherMT, and the final DataUpdateRequestStatusMT will contain
RequestProcessingState set to FAILED.

1.2.10.4 Exchange WEZ
MA will receive the Weapon Engagement Zone (WEZ) through the MS interface. The WEZ
represents a region around the ownship in which an adversary is vulnerable to a shot and is defined
by the range of the weapon system. The WEZ can be utilized in addition to the Dynamic Launch Zone
(DLZ) in the case that there is no target state data, targets have not been assigned, or target state
data lacks fidelity. The WEZ is made up of a Weapon's various ranges. Those ranges are a function
of the weapon type, the azimuth relative to the ownship which the weapon would need to turn to, the
ownship state/position data, and a target's heading, velocity and altitude. The MA_WEZ_RequestMT
message uses the StoreInformation field to identify the specific capability the WEZ is being requested
for. The schema then uses the WEZ_FieldOfView field to identify the region for the WEZ by indicating
Page 92
Mission
Systems
Interface

Mission Systems Interface Volume (v. 5.0a)
the desired azimuth range around the ownship for which the capability ranges are being requested
for. For example, the FRONT_WEZ enumeration represents an array of azimuths from 0 to 180 deg
off the nose of the ownship with a default 5 deg step defining the array. This array of ranges can be
pre-calculated by the service prior to flight or calculated upon the request while in flight. To account
for ownship and potential target state/position data, the implementer has a choice of options that are
outlined by the WEZ_ChoiceType.
The StaticWEZ choice uses the ownship's position data as reported by the vehicle interface and then
the service assumes that the target heading is pointed at the ownship per azimuth outlined in the
WEZ_FieldOfView. The service also will assume the target as the same altitude and velocity as the
ownship. This is the most static, least dynamic representation of the WEZ.
The PlannedStaticWEZ choice allows for the user to specify a planned ownship state/position using
the PlannedReleaseConditions field, The service then also assumes the target's heading is pointed at
the ownship per azimuth but assumes that its velocity and altitude is the same as what is identified in
the PlannedReleaseConditions field.
The PredictedWEZ choice allows for a more dynamic WEZ and requires the user to input the target
state/position data using the PredictedTargetStateData field. This allows the user to input a specific
velocity, altitude, and target heading aspect off the ownship per azimuth. The service would then
subscribe to the ownship state/position data as reported by the vehicle interface.
Lastly, the PlannedPredictedWEZ choice, allows the implementer to specify the ownship and
predicted target state/position data using both the PlannedReleaseConditions and
PredictedTargetStateData fields.
This WEZ request is then acknowledged by MS using MA_WEZ_RequestStatusMT. With the store
information and required inputs for a capability range provided by MA, MS can now send MA the
WEZ specific to that store. MS does so by providing the MA_WEZ message in which provides a
various set of ranges per azimuth requested, which builds out the requested 2D region. The required
ranges that define the WEZ are RangeMinimum, RangeTurnAndRun, RangeProbabilityOfIntercept,
RangeMaxGuidance, RangeOptimal, and RangeMaxAero. The WEZ UCI message structures are
very similar to that of the DLZ messages with the difference being WEZ does not require target state
data

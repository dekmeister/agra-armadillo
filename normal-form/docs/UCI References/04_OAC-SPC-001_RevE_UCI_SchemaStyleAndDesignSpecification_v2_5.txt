Abstract

Universal Command and Control (C2) Interface (UCI) Standard defines a common open messaging approach for mission systems to achieve the business objectives of reducing acquisition and life-cycle costs as well as reducing the risks associated with development, sustainment, technology refresh, and upgrades of mission systems. This specification describes the strict UCI CERTs, GUIDs, and verification procedures for updating the UCI Schema. It is intended to promote consistency across users of the UCI Standard. This document contains two types of design restrictions: CERTs and GUIDs. A CERT is a requirement on the UCI Schema that is strongly enforced and is verifiable using XPath queries or programmatic methods. A GUID is a guideline that while not strictly enforced should be considered during schema development to promote consistency across UCI Schema Adopting Programs (APs). A deviation from a GUID might require sufficient justification to be enacted in the schema. Adherence to the UCI standardized message definitions enables future technology upgrades, providing flexibility that allows providers to offer innovative and competitive solutions to future problems.

Introduction

This UCI Schema Style and Design Specification provides guidance and requirements for developing and evolving the UCI Schema in the UCI Standard. This specification introduces the certifications requirements (CERTs) and Guidelines (GUIDs) that govern the representation of message content in the UCI Schema. CERTs are requirements that are strongly enforced. GUIDs are not strictly enforced but are intended to provide guidance to developers to promote UCI consistency.

UCI, the UCI Standard, the UCI Schema, and the UCI Normalized Interface Specification

Universal C2 Interface (UCI) is a messaging standard for inter-component, mission-level command and control systems. UCI defines a set of messages intended to standardize interfaces between software services for C2, and these messages have been defined and validated as fit for purpose across a variety of mission behaviors and heterogeneous mission system implementations. UCI continues to evolve, and today represents the interactions between mission system software and sensor subsystems to include the full scope of interactions between those components in a standardized, interoperable manner, supporting un-manned and manned systems, in all domains.

The UCI Standard defines UCI, and as depicted in Figure 1.1-1, Elements of the UCI Standard, the UCI Standard is comprised of four elements:

- The UCI Standard Document (OAC-STD-001)

- The UCI Schema (OAC-STD-002), which is released as a zip file

- The UCI Schema Style and Design Specification (OAC-SPC-001) (i.e., this document).

- The UCI Normalized Interface Specification (OAC-SPC-002).

Figure 1.1-1 Elements of the UCI Standard

The UCI Standard Document (OAC-STD-001) defines UCI, providing requirements for UCI Message creation/modification and use, as well as defining UCI governance, change process, and versioning. The UCI Standard Document references this specification to provide both requirements and guidance relating to UCI. The UCI Standard Document does not specify format or transport protocols.

The UCI Schema (OAC-STD-002) is the data model that defines UCI Messages using eXtensible Markup Language (XML) Schema Definition (XSD). The XSD ‘schema’ files describe the UCI Messages. In the context of the CERTs and GUIDs, “UCI XSD Schema” refers to the XSD Schema file “UCI_MessageDefinitions_vM_N_Px_E.xsd” and “UCI_SecurityMarkings_vM_N_Px_E.xsd” where vM_N_Px_E is a version release indicator for a specific UCI Release that is subject to the CERTs and GUIDs documented in this specification in which M is an integer that represents the major release version, N is an integer that represents the minor release version, P is an integer that represents the patch version (e.g., errata), x is one or two lowercase letters that represent an optional engineering release designator, and E is a string that represents an optional 1-45 character extension.

This specification provides CERTs to define how one may construct UCI Messages and data types within the UCI XSD Schema to maintain the common message definition. This approach allows disparate platforms to openly communicate via a system-agnostic set of standardized messages. This specification describes additional restrictions specific to the UCI Standard using terms similar to those defined in the W3C's schema definition specification. The idea is to reduce ambiguity and facilitate ease of understanding.

The UCI Normalized Interface Specification (UNIS) (OAC-SPC-002) provides a set of requirements for normalized UCI message primitive interface behaviors. Normalization of message behaviors and patterns, within UCI, is a benefit to the community as it increases reuse, simplifies integration, increases the ability for the Adopting Program to interact and communicate with other UCI-enabled platforms, and reduces the cost of implementation.

Scope 

This specification defines the CERTs and GUIDs established for UCI Version 2.0 and beyond. UCI CERTs are explicit requirements verified using XPath queries or programmatic methods using a tool like the Schema Checker Tool. UCI GUIDs are general principles or advice used to enhance consistency in messages. This document describes design CERTs and GUIDs, message naming conventions, messaging structure formats, and units and data types used in the UCI Schema. Inherited changes from previous versions of the standards may not fully comply with the CERTs and GUIDs outlined in this specification.

Design Principles Guiding UCI 

A number of guiding principles and assumptions were followed during the development and evolution of the UCI Standard. Some have been codified into the formal CERTs and GUIDs presented in Section 3, while the design principles below are meant to provide guidance during the modification of the UCI Schema and are not considered inviolate. They should be followed to the maximum extent possible. Deviations from these principles should be documented in the corresponding Change Request.

The design principles are as follows:

- Support a loosely coupled SOA with publish/subscribe connections between software services or applications

- Encourage software interoperability by developing thorough, standard message interactions and document them in an XSD file

- Focus the schema on abstract, interoperable representations of data and C2 functions independent from specific program of record implementations; strive for independence from specific program of record architectures, middleware solutions, and messages transports

- Don’t tie the standard to specific technologies

- In the schema, favor elements over attributes

- In the schema, create re-usable simple or complex types for elements or groups of elements that occur more than once

- In the schema, create distinct messages, each as a top-level element, intended to be sent as a whole

- In the schema, balance the desire for fully developed messages with the desire for interoperability by using optional elements

- Decompose messages based on the projected design of functions just enough to support a spectrum of implementations while maintaining interoperability

- Achieve a separation of concerns that supports a wide range of UCI uses/users and domains including ground-based C2 systems, airborne systems, common software services, or applications, modular avionics, etc.

- Create messages to support reasonably granular functions; don’t group multiple functions into one message

- Design messages to support application functions, not application-specific messages

- Don’t assume other implementations will have similar system architectures

- First person information and third person information are different; while the datatypes may be identical, the source of the data may warrant a separate message type

- If there is an existing way to represent data, don’t introduce a new representation without good cause

- Decompose messages whose parts would reasonably be produced at different times for the same object; decouple the corresponding applications or application functions.

These principles are to be followed for any new UCI Message constructs, as well as for revision to any existing messages.

UCI CERTs and GUIDs 

This section describes the CERTs and GUIDs for UCI Schema design. The CERTs and GUIDs apply only to the UCI XSD files. The CERTs are verified using XPath queries or programmatic methods. The section also presents supporting or related GUIDs that represent good practice when modifying existing or introducing new content. While GUIDs are not required to be enforced, they should still be considered during schema development. If deviation from a GUID is required, sufficient reason/rationale should be captured such that appropriate exceptions can be documented in this specification. The UCI Schema is to be program, company, platform, architecture, and transport agnostic. This is required to keep the UCI XSD Schema flexible and usable over a wide range of environments.

XPath expressions are provided over an olive-green background. These can be used to filter content within the UCI XSD Schema definition to verify that a given CERT has been followed. For example, the following expression checks for prohibited attributes defined in the UCI XSD Schema.

The XPath expressions provided in the UCI CERTs can be used to check for individual CERT compliance in many schema development tools such as XML Spy, Liquid XML Studio, and Oxygen.

Schema Standard and File Structure

The CERTs and GUIDs covered in this section are related to how the UCI XSD Schema File is organized and the associated XML standard it has adopted. Enforcement of the CERTs are performed using programmatic methods.

  GUID SCH-000156 [The schema definitions in the UCI XSD Schema should follow version 1.0 of the World Wide Web Consortium (W3C) XML Schema Definition (XSD) Standard.]

  CERT SCH-000157 [The UCI XSD Schema shall follow version 1.0 Extensible Markup Language (XML).]

  CERT SCH-002403 [The xs:schema “version” attribute shall be formatted in accordance with the defined type uci:UCI_SchemaVersionStringType.]

Version 1.0 of the W3C XSD standard has broad industry acceptance while version 1.1 currently lacks broad industry adoption and tool support. The following is an excerpt portion from the UCI XSD Schema. CERT SCH-002403 is to ensure proper formatting of the schema version.

[]

  CERT SCH-000164 [The file that contains the UCI XSD Schema declaration shall be organized into three sections in the following order: global xs:element declarations, global xs:complexType declarations, and global xs:simpleType declarations.]

Group each new element or type with the existing items of the applicable section. The imports and includes (if present in the UCI XSD Schema) must be at the top of the UCI XSD Schema file. Organization of the sections may be verified visually or using programmatic methods.

This makes it easier to locate message elements or types. Alphabetization of the sections may be verified using programmatic methods.

Naming Requirements and Conventions

The CERTs and GUIDs covered in this section relate to the requirements and naming conventions that have been established as part of the UCI standard. Enforcement of the CERTs include either XPath definitions or performed using programmatic methods.

  CERT SCH-000215 [All xs:element entries in the UCI XSD Schema shall be named.]

Anonymous elements are prohibited. Elements that are not named will cause XML binding tools to create their own identifiers which can change whenever the binding tool is re-run. This CERT SCH-000215 can be tested using the following XPath.

//xs:element[not(@name)]

  CERT SCH-002404 [All xs:element entries within a complexType including those which extend from other types shall be uniquely named.]

Element names within a complexType including those which extend from other types, should not have duplicate names since these will also cause XML binding tool issues. The CERT-SCH-002404 can be tested using programmatic methods.

  CERT SCH-000220 [All xs:complexTypes and xs:simpleTypes in the UCI XSD Schema shall be globally defined.]

Anonymous types are prohibited. This prevents XML binding tools from creating arbitrary names when code is generated. This includes bounded lists declared with no types.

The following example is prohibited.

[]

Instead, this would be defined as follows.

[]

This CERT SCH-000220 can be tested using the following XPath.

//(xs:simpleType | xs:complexType)[not(parent::xs:schema)]

  CERT SCH-000262 [A message type declaration shall be specified by using the name of the associated message followed by “MT” (<MsgName>MT).]

  CERT SCH-000263 [A message data type declaration shall be specified by using the name of the associated message followed by “MDT” (<MsgName>MDT).]

The underscore is included as part of <MsgName> when <MsgName> ends with an abbreviation. See CERT SCH-000491 for more details. CERT SCH-000262 can be checked using the following XPath.

//xs:schema/xs:element[ not(ends-with(@type,concat(@name,"MT")) or ends-with(@type,concat(@name,"_MT")))]

CERT SCH-000263 is enforced using programmatic methods.

  CERT SCH-000272 [For every declared global element (Message), a valid schema shall contain a message type (“MT”) and message data type (“MDT”) declaration that are associated with the Message.]

  CERT SCH-000273 [Each Message name shall appear in the uci:MessageTypeEnum.]

  CERT SCH-000274 [Each enum value in uci:MessageTypeEnum shall correspond to a Message name.]

Each message defined as a global element must have a corresponding MT and MDT type defined. An example to the naming convention is shown in the table below. These CERTs are enforced using programmatic methods by evaluating the name of messages against contents of the uci:MessageTypeEnum and expected complexType MT and MDT declarations. Message names added to the uci:MessageTypeEnum are formatted using underscore separation between each upper camel case segment of a message name. For example, an entry in uci:MessageTypeEnum for AirSampleActivity would appear as AIR_SAMPLE_ACTIVITY. Additional examples are provided below.

  --------------------------------------------------------------------------------------
  Message Name        MT                    MDT                    MessageTypeEnum
  ------------------- --------------------- ---------------------- ---------------------
  AirSampleActivity   AirSampleActivityMT   AirSampleActivityMDT   AIR_SAMPLE_ACTIVITY

  AMTI_Activity       AMTI_ActivityMT       AMTI_ActivityMDT       AMTI_ACTIVITY

  ApprovalAuthority   ApprovalAuthorityMT   ApprovalAuthorityMDT   APPROVAL_AUTHORITY
  --------------------------------------------------------------------------------------

  CERT SCH-000297 [All message types shall correspond to a global element (message).]

  CERT SCH-000298 [All message data types shall correspond to a global element (message).]

MT and MDT types must only be defined for a particular global element (message). If they exist without a corresponding message, then they have been abandoned and should be removed. Similar to SCH-000272, SCH-000273, and SCH-000274, these CERTs are enforced using programmatic methods by evaluating declared MT and MDT types against the set of declared messages in accordance with the naming conventions.

  CERT SCH-000300 [The names of complexTypes shall end with "Type", "MT", "MDT", "DRL", "DRLE", "ML", "PET", or "EXT".]

Naming conventions for complexTypes require all types to have a name ending with: “Type”, “MT”, “MDT”, "DRL", "DRLE", "ML", “PET”, or “EXT”. See Section 4.3.1 ComplexType Construction, for details on "DRL", "DRLE", "ML", “PET”, or “EXT”. The “Type” is used if none of the other strings apply. The CERT SCH-000300 can be tested, using the following XPath.

//xs:complexType[not (ends-with(@name,"Type") or ends-with(@name,"MT") or ends-with(@name,"MDT") or ends-with(@name,"DRL") or ends-with(@name,"DRLE") or ends-with(@name,"ML") or ends-with(@name,"PET") or ends-with(@name,"EXT"))]

  CERT SCH-000305 [A type derived from uci:ID_Type shall have a name ending with “ID_Type”.]

Identifier types containing a Universally Unique Identifiers (UUID) are a construct adopted by UCI to identify items referred to by the UCI XSD Schema. For consistency, identifier types containing a UUID are complexTypes that will always have a name ending with “ID_Type” and will always extend the established base type uci:ID_Type.   CERT SCH-000895 [Bounded lists of enumerated types shall have a maxOccurs equal to the number of items declared in the enumeration (“Select All That Apply”) or unbounded (“An Ordered List of Enumerations”).]

To understand how bounding rules should be applied to lists of enumerations, one has to understand how such lists are used. There are two known use cases that are applicable to the use of lists of enumerations: “Select All That Apply” and “An Ordered List of Values”.

- Select All That Apply: This use case involves the selection of all applicable values from a set of values. Such a selection entails that there are no duplicates of enumerated items in the selection. An example of this use case is where a service would like to receive a subset of status information from a service capable of reporting a large variety of status information. This specification would allow the requesting service to request only the information it requires. For this type of usage, a list of enumerations is used where the size of that list equals the number of items in the enumeration. In other words, the bounds for such a data type declaration would be specified as minOccurs=”1” and maxOccurs=”N” where N is the number of items in the enumeration. Note that for most uses that fall into this category the specified list of enumerations is generally considered to be unordered and that no interpretation should be applied to the order with which the enumerations are specified. Annotations should indicate 'Select All That Apply' as the rationale to justify the bound.

- An Ordered List of Enumerations: This use case involves specifying a list of enumerations in which the order in which the values are specified is to be taken into account when processing the list. In addition, unlike the previous case, there may exist duplications in the items selected. An example of this use case is where a set of operations is specified to be performed by a service, in the order in which those operations are to be performed. Also, operations to be performed may be repeated any number of times. For this type of usage, a list of enumerations is used where the size of that list is unbounded. In other words, the bounds for such a data type declaration would be specified as minOccurs=”1” and maxOccurs=”unbounded”. Note that for a specific program or domain, it may be possible or even necessary to place an upper bound on such a list. However, in general, specifying such an upper bound is not possible.

This CERT is enforced using programmatic methods by evaluating if a non-unbounded value to a list using enumeration types equates to the total number of values in the enumerated list.

  GUID SCH-000964 [Bounded lists of non-enumerated types in the UCI XSD Schema should have maxOccurs set to unbounded.]

All bounded lists of simple enumerated types must adhere to CERT SCH-000895. Consideration, however, for bounded lists of complex types should consider cases as outlined below. For all other cases where elements of complexType or simpleType (non-enumeration), unbounded multiplicity is recommended in the UCI XSD Schema. Multiplicity of any element where a specific definition cannot be consistently applied between all uses should be left unbounded. Do not create arbitrary bounds.

However, bounding, in general, can be used where strategically possible. Multiplicity of an element that is either of complexType or simpleType (non-enumeration) can establish a defined boundary so long as it can be applied consistently between all uses. Where bounded settings are established, an appropriate annotation should be added to capture the rationale for defined bounding conditions (e.g., could be based on external standards such as MIL-STD-6016, NITF 2.1, etc.).

UCI Schema Structure

This section describes the UCI Schema Structure, which characterizes the content and organization of the UCI XSD Schema. It starts with the basic organization of the file. Then it introduces the basic contents of a message and its basic components. Next it introduces the standard building blocks: complexTypes and simpleTypes. Here, examples of the UCI XSD Schema structure patterns and uses are presented that reflect the importance of GUID SCH-000900 [SCH-000900]. This section concludes with guidelines for annotations and their importance to the UCI XSD Schema. Throughout this section, CERTs and GUIDs that are applicable to particular content presented are referenced at the end of each associated sentence with one or more references with use of “brackets” ([]) and appropriate CERTs or GUID identifier [SCH-######].

File Structure

The UCI Schema consists of three basic sections [SCH-000164]: Global Elements (also known as messages), Global complexTypes, and Global simpleTypes. Each section is organized alphabetically for readability purposes [SCH-000166]. The Global Element section contains the formal declaration of all messages. Global simpleTypes are the basic building blocks – enumeration, numerical, and string types that are utilized in constructing various complexTypes. Global complexTypes are the higher-level data types constructed from lower-level simple and lower complexTypes. These types are key building components for UCI Message constructs. The organization of the UCI Schema XSD file further reflects the Venetian Blind Design approach the UCI Standard follows by allowing for the decomposition of types into reusable complex and simple type definitions promoting maximum reuse in the UCI Standard [SCH-000168, SCH-00961].

Basic UCI Schema Message Structure

Like any message standard, there is normally an established infrastructure. UCI is no different in establishing such an infrastructure. For UCI, the basic structure of any message consists of two parts: Message Header and Message Body. Figure 4.2-1, Basic Message Structure, describes the basic message structure in UCI.

- The Message Header is divided into two sections. The first part is the security information where UCI uses a complexType SecurityInformationType that mirrors security markings that map directly to the IC-ISM Data Encoding Standard. The second part of the header consists of UCI Specific Data.

- The Message Body consists of two parts: an Object state (ObjectState minOccurs=0) in accordance with Table 3.5-1 Required Message Type Elements for Each Primitive Type and a Message Data segment. MessageData is the portion of a message that is unique to a particular message. If this Message Data segment contains a distinct object for the message, this object is expected to be tracked and managed across multiple objects or messages. This is accomplished for a specific message using ObjectState and a unique Object ID (based on the specific message name) located in its Message Data segment along with its associated object data. (See Section 5.0 for further definition of an object.)

The remaining subsections in 4.0 will describe the construction of lower level complexTypes and simpleTypes that are used to build the contents of these segments. Section 5.0, Message Definition Patterns, captures the message definition and patterns common in the UCI Schema.

As stated above the SecurityInformationType in UCI maps well to the IC-ISM markings. This is intentional as it allows those who wish to create a UCI encoding that conforms to the IC-ISM Data Encoding Standard with a simple mapping. Because the IC-ISM schema uses attributes extensively and attributes are discouraged for UCI, an effort was made to ensure that the portion of the IC-ISM Standard could be mapped well to constructs within UCI that meet this style specification. Version 2021-NOVr2022-NOV of the IC-ISM Standard was used for this mapping.

Message Data Type Construction

The message data type in each message is the primary building block that contains the data content necessary to achieve the message’s purpose. As the foundation in a message, the MDT is built using the xs:complexType tag. The construction is achieved by decomposition of required data content into lower-level types that aligns with typical interfaces and levels of control [SCH-000902]. The fundamental building blocks used consist of lower-level forms of complexTypes and the basic representation of data in either primitive or simpleType forms. Construction of these types are described further in Sections 4.3.1, ComplexType Construction and Section 4.3.2, SimpleType Construction. Schema additions should use or extend existing types (Complex and Simple) where practical.

ComplexType Construction

The UCI Schema has established a variety of construction patterns for all named complexTypes [SCH-000491, SCH-000220, SCH-000505]. Section 4.3.1.1 describes the two common complexType construction patterns used in UCI: sequence types and choice types. UCI has also established a set of specialized complexType construction patterns. These include identifier (ID) types (Section 4.3.1.2), abstract types (Section 4.3.1.3), and recursion types (Section 4.3.1.4).

Basic ComplexType Structures

There are two basic types of complexTypes used in UCI XSD: sequence based, and choice based [SCH-000564]. These two basic types are constructed using xs:element tags with accompanying descriptive annotation (xs:annotation) [SCH-001051]. Complex types shall only specify the following attributes: name, abstract, version [SCH-009995]. Elements within an xs:complexType shall only specify the following attributes: name, type, minOccurs, maxOccurs [SCH-009996]. Other constructive component building blocks (ref, xs:attribute) are prohibited [SCH-000564, SCH-000590]. As with messages, all complexTypes are required to be named with specific guidelines that is not a programming language (C++, Java) keyword [SCH-000220, SCH-000505, SCH-000423]. ComplexTypes, once defined, are also required to be annotated and used [SCH-001044, SCH-000800]. An allowance to the aforementioned required use is extended to the special class of complexTypes which extend from a PET ComplexType. Also, all fields (elements) within a complexType are to be named along with an existing named type [SCH-000220].

The basic sequence complexType structure is shown in Figure 4.3-1, Basic complexType Sequence Declaration in Schema. The use of xs:sequence does not use attributes in UCI [SCH-003081]. Here, an example complexType, named BasicSequenceType, consists of three elements: Two elements using lower level complexTypes and one element using a lower level simple type. As shown, all fields in the complexType use an xs:element tag and include an xs:annotation [SCH-000744, SCH-001044]. When creating a complexType, the UCI Standard promotes maximum reuse with type definitions as the main form of component declaration [SCH-000168, SCH-000961]. While the UCI Standard promotes reuse, it discourages duplication of types to reduce bulk and maintenance and does not allow for unused types to exist in the UCI XSD Schema with allowance to the special class of PET types [SCH-000961, SCH-000800].

Figure 4.3-1 Basic complexType Sequence Declaration in Schema

The basic choice complexType structure is shown in Figure 4.3-2, Basic complexType Choice Declaration in Schema. Here, an example complexType, named BasicChoiceType consists of three elements: two elements using lower-level complexTypes and one element using a lower-level simple type. As shown, all fields in the complexType use an xs:element tag, are named, are required (minOccurs=1, default), use a valid named type, and are annotated [SCH-000220, SCH-000744, SCH-000651, and SCH-001044]. The choice type requires a name but does not support the use of attributes such as minOccurs or maxOccurs [SCH-000651]. A choice type also cannot be extended from an existing type, unless that base type is abstract and contains no elements [SCH-000685, SCH-000687]. There are other forms of complexType construction patterns in UCI. These types are identified and described further in Sections 4.3.1.2 through 4.3.1.5.

Overall, the family of complexTypes have different functions within the UCI Schema. Certain behavior or purpose can be inferred by the string characters at the end of the name. As stated earlier, all complexTypes ending in “MT” are directly related to the messages constructed in the UCI XSD Schema, complexTypes with MDT associated with a particular message data component. Other specialized types (BaseType, PET, EXT) are described in Section 4.3.2.3. Abstract Types. All others that do not fall into anyone of these will end with “Type” [SCH-000300]. UCI does not allow for other forms of complexTypes which include mixed types, anonymous types, the use of simpleContent, and use of complexContent with restrictions [SCH-000564, SCH-000769, SCH-000775, SCH-000781]. UCI has also established a pattern of class of unique identifier types. These are further described in Sections 4.3.1.2, Identifier Types.

Identifier Types

UCI has established a pattern for constructing identifier types. There are three forms that exist within UCI: Universally Unique Identifiers (UUID), Foreign Key, and Index.

UUIDs are used when needing to establish a unique context, i.e., when it is necessary to uniquely identify an object or reference an object from another message. All UUID-based types extend uci:ID_Type that consists of a UUID and a descriptive label [SCH-000870]. Some UUID-based types extend the uci:ID_Type and include an additional element for version identification. As part of a naming convention, the names of UUID-based identifier types end with the “ID_Type” [SCH-000305]. In addition, the names of all elements using an identifier type ending with “ID_Type” will have a name ending with “ID” [SCH-000310, SCH-000311]. Figure 4.3-3, Base ID_Type, is the schema base type for all derived UUID types in the UCI Schema. Figure 4.3-4, Based Version ID_Type, is the version based id type that directly derives from the base “ID_Type”.

Abstract Types

Besides the basic sequence and choice types, there exists other fundamental complexType constructs used in UCI. These complexTypes are better known as abstract types. Just as abstract classes in software representation are not designed to be instantiated, the same rationale in using the abstract attribute in the UCI Schema is to prevent instantiation in software of these designated types. For example, if the intent of a type is to establish the minimum requirements with a family of types, then the abstract attribute for this type would be set to true. The use of abstract types has multiple forms in UCI. There is the common “BaseType” that establishes a basic construct that higher level types can derive from. A special case of this kind was introduced earlier – namely uci:MessageType. Other similar families of types exist in UCI and whose name commonly ends with “BaseType” [SCH-000511]. Details of the use of these families of base types will be discussed later in Section 5.0, Standard Message Patterns [ SCH-000511, SCH-000512, SCH-000513].

Two other abstract family types also exist [SCH-000512, SCH-000513]. These two are Extension Types (EXT) and Polymorphic Extension Types (PET). EXTs are abstract types with no content that serve as an extension point for types that cannot be included in the unclassified standard. PETs are abstract types that are intended to represent a related set of types and follow a naming convention that allows for identifying a relationship to the associated family. An xs:extension shall only specify the following attributes: base [SCH-009997]. Figure 4.3-6, Basic Polymorphic Extension Type (PET) Declaration in Schema, provides an example of a PET and associated naming convention.

[]

Figure 4.3-6 Basic Polymorphic Extension Type (PET) Declaration in Schema

With PETs, an alternate method for using a choice type is available through the use of Polymorphism in UCI. With a controlled set of complexTypes, defined elsewhere in the UCI XSD Schema, an element declared using a PET type allows access to all derived types. PETs have also been used in UCI to address support for multiple versions of formatting standards such as NITF or STANAG 4607 by allowing the overloading of options for different versions. Two examples are NITF_PackingPlanPET and STANAG_4607_PackingPlanPET.

Recursion Types

Per Design GUID SCH-000761, “A valid schema should not contain any cyclic dependency graphs.” While UCI discourages the implementation of potential cyclic dependencies, there are cases where known potential cyclic dependencies exist. Several exceptions were referenced within GUID SCH-000761.

Another form of potential cyclic dependency in the UCI Schema is specialized self-reference, used for recursion. There are complexTypes that are designed to allow limited recursive behavior during message construction. As previously illustrated in GUID SCH-000761, Figure 4.3-7, General Example with Recursive Run-Time Behavior, depicts general run-time behavior of a complexType with recursion. To implement such behavior in the UCI Schema, Figure 4.3-8, ComplexTypes with Recursive Behavior, provides the guidance in implementing a recursive behavior in UCI. The permitted structure for recursion is a choice type with two elements: One element is an abstract type, such as a PET, which is the point where recursion occurs in an implementation, and the other element is a terminator of type uci:EmptyType to indicate the end of recursion. Note that the recursive nature of the PrefixBasicRecursiveType is evident with the use of the BasicRecursivePET in the PrefixBasicRecursiveChoiceType. Also, note the required element called “Terminator” that uses the uci:EmptyType in providing a termination indication for recursive behavior [SCH-000789, SCH-000790, SCH-000791].

An example in the UCI Schema exists within the message StoreLoadoutConfiguration where the StoreLoadoutItemType type implements the RecursivePET and Terminator pattern. See the UCI Schema for more details.

Another message that uses recursion at runtime is QueryDataRequest, which has a QueryPET element that is generally replaced by the concrete extension QueryType. Since QueryType itself has elements of type QueryPET, this creates a self-reference that typically includes a terminator per GUID SCH-000761. The lack of a Terminator in QueryType is an allowable exception to this guideline.

Patterning of Kinematic Types

Kinematics is the study of motion without regard for its cause. General use of kinematics in UCI describes motion by establishing a reference frame (Earth-centered, Earth-fixed (ECEF), World Geodetic System 1984 (WGS84), etc.) used to define location of an object and its associated motion using two or more dimensions. Additional characteristics may also include uncertainty ellipse for the motion terms, covariance, and orientation terms.

Figure 4.3-11, Basic Kinematic ComplexType Pattern, describes the typical makeup of most kinematic types constructed in UCI [GUID SCH-003266]. Here, the usage is illustrated by the fields’ outlines and connections. Fields with solid outlines must be included in all kinematic type structures. Conversely, fields with dashed outlines may be included in (or omitted from) kinematic type structures depending on the type’s intended use case(s). When a field is included in a kinematic type’s structure, its connections indicate whether it will become a mandatory (solid lines) or optional (dashed lines) element in UCI schema.

[]

Figure 4.3-11 Basic Kinematic ComplexType Pattern

The following subsections further elaborate on the basic characteristics defined above. Following the subsections defining the kinematics structure, the current UCI kinematics types are listed with their respective alignment to the kinematic type pattern.

Timestamp

All kinematic types include a mandatory timestamp of type uci:DateTimeType as shown in Figure 4.3-11, Basic Kinematic ComplexType Pattern. This timestamp is the primary time reference for valid data included within the kinematic type. Any other timestamps that exist within the type should be limited to supplemental origin or kinematic information and be clearly annotated to indicate purpose and use.

Local Origin Definition

All kinematic types will include a defined local origin term from which location with motion information is conveyed. This mandatory term is not necessarily the origin of the coordinate system; it can also be a reference point in a coordinate system (e.g., WGS84) or coordinates of a local origin in the base reference frame (e.g., ECEF). The origin should consist of three component elements (e.g., Latitude, Longitude, Altitude) and be annotated with appropriate units of measure (radians, meters, etc.) as shown in Figure 4.3-12, Local Origin Definition Details. Optionally, the reference point could include a choice series including the Origin Coordinates along with one or more external reference(s) such as a ReferenceFrameID or ECEF Position.

When defining the origin, frame transformation matrices may be utilized for converting coordinate data between different reference frames, such as from a Local NED system to an ECEF system. Frame transformation consists of a translation followed by rotation. Some examples of use cases include precise navigation, geolocation, or data fusion from one or more sensors. If desired, include an optional element named “FrameTransformation” using the type FrameTransformType. In most cases, a single instance of frame transformation meets the needs of a given type. However, some cases warrant additional frame transformation operations. If required, the pattern allows for multiplicity of this element. When introducing a new coordinate system, an enumeration reflecting the new coordinate system (e.g., ECEF, NED) must be added in TransformFrameEnum.

[]

Figure 4.3-12 Local Origin Definition Details

Along with the core parts defining the origin, additional supplemental origin information can also be included that supports other functional needs. This provides a method for tailoring while still conforming to the general pattern. This supplemental origin information should be captured in its own unique type in the schema. Some examples of data may include, but is not limited to, support for altitude offset or correction parameters.

Kinematic Information

Along with the origin, the basic kinematic type structure includes the corresponding kinematic information as shown in Figure 4.3-11, Basic Kinematic ComplexType Pattern. Kinematic information is provided using four main components, location with motion, covariance matrix, orientation information, and, optionally, supplemental kinematic information. The structure of this information is provided in this section along with guidelines for assembling new types.

Kinematic Information: Location with Motion

The mandatory location with motion term is defined using three-dimensional vectors (3D-vectors) describing the location and, optionally, velocity and acceleration (see Figure 4.3-13, Kinematic Information: Location with Motion). Common examples for 3D-vectors defining a location include:

- Global coordinates (i.e., ECEF coordinates)

- 3D-vectors configured relative to sensor line of sight (e.g., line of sight (LOS), 3D line of sight (LOS3D), or cone angle) with respect to a local coordinate system (the local frame is, in turn, defined with respect to a global coordinate system).

- Local 3D-vectors configured with respect to a global coordinate system (i.e., North/East/Down).

[]

Figure 4.3-13 Kinematic Information: Location with Motion

With each 3D-vector element, an associated uncertainty ellipse may be included. If included, this element is set to optional in the schema. Uncertainty ellipses provide specific error estimates following a simpler format than covariance matrix. They provide a specific accuracy (e.g., within a stated circular error probability/elliptical error probability, CEP/EEP) such as for tactical displays, mission planning inputs, or conveying error with specific vector measurement. When included, any new structures describing the associated ellipses should align with current structures defined in the schema (Figure 4.3-13, Kinematic Information: Location with Motion). Uncertainty ellipses are defined consistently between coordinate frames via the following components:

- Ellipse: Lengths of ellipse’s semi-major and semi-minor axes

- Orientation: The angle between the semi-major axis and the x-axis.

- 3^(rd) Axis (choice)

- Vertical Thickness: An uncertainty value orthogonal to the plane of the ellipse. Shape is like a hockey puck

- 3^(rd) Ellipsoid Axis Length: Axis is the cross product of the semi-major and semi-minor axes of the ellipse.

- Tilt: The angle between the plane of the ellipse and the plane orthogonal to the z-axis.

Inclusion of the “Vertical” and “Tilt” fields is mandatory for all kinematic types; inside the kinematic structure, they are optional elements in the schema (Figure 4.3-13, Kinematic Information, Location with Motion).

Kinematic Information: Covariance Matrix

Along with the required location with motion terms, kinematic information can, optionally, include covariance matrix terms (Figure 4.3-14, Kinematic Information: Covariance Matrix). Covariance terms are included to represent errors used for propagating or predicting past or future kinematic states. If included, the kinematic type would include an optional element “CovarianceMatrix” using the type “CovarianceMatrixType”. This type consists of three elements, an optional “TimeBiasError” of type “xs:duration”, a “Category” of type “CovarianceMatrixCategoryEnum”, and a “Covariance” list of type “xs:double”.

[]

Figure 4.3-14 Kinematic Information: Covariance Matrix

The “TimeBiasError” represents the 1-sigma error in nanoseconds as defined in the corresponding annotation. The “Category” specifies the covariance matrix being represented based on the value selected from the enumerated type “CovarianceMatrixCategoryEnum”. This simpleType contains enumerations that are named to identify available covariances. These enumerations contain supporting annotations that define size and term order of the “Category” selected. The selected enumeration governs the structure of the “Covariance” list. All covariance lists describe the lower triangular terms contained in the matrix. The list of values is ordered based on the row-ordered lower triangular half of the matrix from element 1,1 (item 1) through the last element used in the correction. For example, the enumerated value POSITION_VELOCITY_ACCELERATION represents a covariance matrix where, based on the name, the first three diagonal terms represent position (ex: ECEF (x,y,z), Px/Py/Pz), followed by velocity, and then acceleration. Included with the enumerated value is this example corresponding annotation:

A 9x9 symmetric covariance matrix where the diagonal terms are ordered starting with Position (ex: ECEF (x,y,z), Px/Py/Pz), followed by Velocity (ex: Vx/Vy/Vz or Vn/Ve/Vd) and then Acceleration (ex: Ax/Ay/Az or An/Ae/Ad). The list of 45 values correspond to the row-ordered lower triangular covariance matrix terms.

The annotation defines the size of covariance matrix, prescriptive order of covariance terms based on the diagonal term order and the overall list size. With the enumerated value selected and supporting annotation, the corresponding “Covariance” terms results in a row-ordered lower triangular list of values of the covariance matrix as shown in Figure 4.3-15, Covariance Matrix Example (POSITION_VELOCITY_ACCELERATION). Any new enumerated values required should adhere to the enumerated value naming convention pattern and its supporting annotation needs.

Finally, in most cases a single covariance matrix meets the needs for the corresponding kinematic type. However, some cases warrant additional covariance matrix needs. If required, the pattern expects the implementation to update the multiplicity of this element with an appropriate annotation describing this need.

[]

Figure 4.3-15 Covariance Matrix Example (POSITION_VELOCITY_ACCELERATION)

Kinematic Information: Orientation/Attitude Information

Along with describing location with motion, the kinematic information can, optionally, include information describing the orientation of the object. The location with motion provides kinematic information treating the object of interest as a point within the kinematic frame. Orientation information supplements the location with motion by describing object of interest’s body frame with respect to the object’s kinematic frame. The combination of location and orientation fully describe how the object is placed in space. The terms include a required term describing the orientation of a rigid body with respect to a fixed local coordinate system applied to define location and may include optionally it’s 1st and 2nd time derivatives. Two forms of describing orientation with motion are available, Euler Angle based (OrientationType) and Quaternion based (QuaternionType) (Figure 4.3-16, Kinematic Information: Orientation). If time derivative terms are included, the terms should be optional in the schema.

[]

Figure 4.3-16 Kinematic Information: Orientation/Attitude

Kinematic Information: Supplemental Kinematic Information

Along with the core parts defining the kinematic information, kinematic types may include optional supplemental kinematic information to support other functional needs (see Figure 4.3-11, Basic Kinematic ComplexType Pattern). This provides a method for tailoring while still conforming to the general pattern. As within the origin supplemental definition type, this supplemental information should be captured in its own, unique type and its use be set to optional in the schema. Some examples of data may include, but are not limited to, additional velocity terms, wander angle, magnetic heading, or air data terms.

SimpleType Construction 

The lowest level components in constructing complexTypes in UCI consists of allowed xs primitive types, UCI built-in simpleTypes with restrictions on xs primitive types, and UCI simpleType enumeration types. These lower level primitive types and simpleTypes are the building blocks for all UCI complexTypes. The family of UCI simpleTypes consists of a named type whose name ends with one of two options: Type and Enum [SCH-000220, SCH-000362, SCH-000363, SCH-000364]. Sections 4.3.2.1 and 4.3.2.2 provide details on these two types and any established naming conventions and styles commonly used in building these lower level simpleTypes [SCH-000505, SCH-003119, SCH-003120]. SimpleTypes are required to follow specific CERTs and GUIDs for naming and use throughout the UCI XSD Schema.

Enum Types

There is a special class of restricted string types called enumerations. These types are constructed similar to the basic simpleType allowing the definition of a list of possible string values. Enumeration types benefits are represented by the definition of GUID SCH-000959, where enumeration types establish sets of distinct types whose values are restricted to one of several defined constants. As with any simpleType, the basic construct of an enumeration simpleType consists of an xs:simpleType definition tag, a unique name attribute that is not a programming language (C++, Java) keyword and ending in Enum an xs:annotation (see Section 4.3.3), an xs:restriction base of xs:string, and one or more xs:enumeration restriction attributes with corresponding xs:annotation [SCH-000423, SCH-000363, SCH-001037]. Values for each simpleType enumerations shall be unique and follow previously defined restrictions [SCH-000394, SCH-000396, SCH-003472]. Figure 4.3-19, Example of Enumeration Type, provides an example of how enumeration types are constructed in the UCI Schema.

Enum types are used in some instances to provide ways to trigger an action or command or provide a result or response to an action or command issued. One special type “MessageTypeEnum”, includes a reference to each message in the UCI XSD Schema [SCH-000273, SCH-000274]. With this, the message ResendRequest allows for requesting a selected message to be resent based on the value set in the element MessageType. Similar uses of message names exist in other enumeration types. The enum types associated with message outputs provide a mechanism in identifying output behavior associated with an associated capability type. Some examples of these enum types are AirSampleMessageOutputsEnum, AMTI_MessageOutputsEnum, AO_MessageOutputsEnum, EA_MessageOutputsEnum, ESM_MessageOutputsEnum, PO_MessageOutputsEnum, SAR_MessageOutputsEnum, and SMTI_MessageOutputsEnum. For these special enum types, introduction of an enum value into these types for a new message should also considered based on the related capability use case.

UCI has also identified two special categories of enum types that have specific usage in the standard. The two categories are State and Status. To assist in determining how these special terms are included as part of the name of a type, the standard has established definitions to provide a means for assessing the usage of the type as it is introduced and used within the UCI Standard. These definitions are to be used to test the primary usage of the type.

The following provides definitions for the terms State and Status with regards to how they are intended to be used when naming fields, data types, and Messages in the UCI Schema.

Definition for State:

-  The term State is defined as a condition or set of conditions that exist at a specific point in time. States are deterministic and mutually exclusive. A state must have a defined state transition such as a trigger (event causing transition), a guard (condition that must be present for transition), or an effect (function to be performed during transition). States may be commandable. States can be reported as status. SimpleType State enum simpleTypes are designated ending the name with “StateEnum” to the name of the type.

Definition for Status:

- The term Status is used to refer to other, non-state, attributes of an entity as well as the results of an operation, i.e., some action that was performed. Note that here we are referring to an entity in general and not an executing entity specifically. Executing entities are a subset of entities. Also note that an operation may generate intermediate Status information as it is performed.

- An outcome of an operation (e.g., success/fail, activated)

- Status enums are not necessarily mutually exclusive

- Status is not commandable.

- SimpleType Status enum simpleTypes are typically designated ending the name with “StatusEnum” to the name of the type.

Future messages introduced to the UCI Standard must align with these definitions.

Message Definition Patterns

In Section 3.0 of this Specification, the UCI CERTs and GUIDs that govern the construction of complexTypes and simpleTypes in the UCI Schema were established. In Section 4.0, the UCI Schema Structure was described with references to the appropriate CERTs and GUIDs. Now in this Section 5.0, the different UCI Message definition patterns are presented, starting with message primitive patterns and evolving to message group patterns.

UCI has established a set of basic message definition patterns with message and type naming conventions as a basis for creating messages in the UCI Schema. Throughout this section, the basic pattern and naming conventions established by this Specification focuses on UCI Message patterns, their purpose, and the applicable guidelines for building such patterns.

Before discussing the established message primitive patterns and other group patterns, the definition of a message object must be established. The existence or non-existence of an object represented within a UCI Message will be used to establish a message’s primitive pattern.

A UCI message object represents something (e.g., emitter, track, plan, report, weapon, etc.) in the battlespace (or a portion thereof) that has state and characteristics. A message object is a collection of data, including a unique Object ID, contained within a message and it is expected to be managed and referenced in other objects or messages whether or not data record management is employed, and its ObjectState is solely managed within the message that contains the object. Additional details include:

- This message object is established using the Object’s state (ObjectState) and a unique Object ID located in its Message Data segment with its associated Object data

- The name of the Object, its Object ID, and their underlying types align with the name of the message (e.g., Entity message contains an Entity Object of type EntityMDT, which contains an Object ID called EntityID of type EntityID_Type)

- The Object ID is based on the extended ID_Type consisting of a universally unique identifier (UUID) and descriptive label

- The Object ID can be used as a reference to this data in other objects and messages (e.g., EntityID is one example of an object referenced in many other messages)

- The Object’s state (ObjectState) is controlled by the message. While other messages may reference data in another object using the message’s Object ID, those messages do not manage the state of the Object

- Message objects only apply to Data-1 and DataRecord-1 message primitive patterns

- Message objects do not exist in Status-1, Command-2, DataRequest-2, and ActionRequest-2 message primitive patterns.

GUID SCH-000900 references this section for acceptable UCI Message 'styles', other content should be considered as advisory.

  GUID SCH-002837 [A message should adhere to the content and naming conventions established by the corresponding primitive pattern templates captured in Appendix A, UCI Message Pattern Templates.]

  GUID SCH-002838 [When applicable, a message should also adhere to the content and naming conventions established by the corresponding group pattern templates captured in Appendix A, UCI Message Pattern Templates.]

Appendix A, UCI Message Pattern Templates, is included to provide adopting programs guidance on building UCI messages in accordance with known pre-existing message patterns and content of similar types. Usage of these templates are expected as part of any new messages developed and introduced into the UCI Standard. Existing exceptions to names and content have been identified and are grandfathered by this standard. No future exceptions will be allowed.

Message Primitive Patterns

Message primitive patterns are composed of one or two UCI Messages and are used to build more complex message group patterns (see Section 5.2, Message Group Patterns).

There are six (6) message primitive patterns. Three (3) message primitive patterns exist for single messages pertaining to data and status called Data-1, DataRecord-1, and Status-1, respectively. Three (3) message primitive patterns exist for paired messages, ActionRequest-2, Command-2, and DataRequest-2 [CERT SCH-002460 and CERT SCH-002489]. The paired messages have an inherent message that triggers a message response pattern based on one of three types: commands, action requests, and data requests. All six (6) primitive message patterns are described in Table 5.1-1, Message Primitive Patterns, and further defined in the following subsections [GUID SCH-000555]. Note that all patterns are named using the format “<PatternName>-n”, where n is the number of messages in the pattern.

All existing UCI Messages have been mapped to one of these six message primitive patterns. Any new proposed UCI Messages must follow one of these message primitive patterns without exception.

Command-2 Message Primitive Pattern

Any message containing command or command status information with no defined message object follows a Command-2 message primitive pattern. This message primitive pattern follows a command response pattern used by a message sender to command a message receiver to perform a command as soon as possible and the message receiver responds to the message sender immediately with corresponding command status. Other messages will provide output(s) of the executed command.

The Command-2 message primitive pattern involves two messages [CERT SCH-002461]:

- <MsgName>Command

- <MsgName>CommandStatus.

Capability Definition

A capability is a performable action useful in accomplishing missions. Capabilities are typically tasked, discovered, commanded, or any combination thereof, using messages.

Capabilities are advertised in the <MsgName>Capability message, and then the status of each advertised capability is reported in the corresponding <MsgName>CapabilityStatus message. Some capabilities may be further defined into sub-capabilities, which are also advertised and reported in the <MsgName>Capability and <MsgName>CapabilityStatus messages. Sub-capabilities may be employed by the containing capability to execute specific aspects or approaches to accomplish missions.

Sub-capabilities are typically tasked, discovered, commanded, or any combination thereof, only via the capability in which they are provided. A sub-capability is the second tier of the capability taxonomy and is provided only within the context of a capability; it cannot be accessed independent of that capability. A capability can be advertised by a subsystem or service.

A capability has a defined state behavior as shown in Figure 5.2-6, CapabilityAvailability State Diagram, that is reported as Availability in the <MsgName>CapabilityStatus messages. Only the defined state enumerations and transitions are allowed. A defined state or transition is not required to be used, but if a state or transition is supported, it must be used as intended.

Subsystem Definition

A subsystem is comprised of one or more physical components and software application(s) that provide one or more capabilities. Software applications running in a subsystem do not have to be presented as services. SubsystemStatus is used to report a subsystem's current operating state periodically and/or aperiodically (i.e., upon request). A defined state or transition is not required to be used, but if a state or transition is supported, it must be used as intended. Some subsystems may power up directly into the OPERATE state and bypass the other defined states.

Capability-7 Message Group Pattern

Glossary

The following are terms used in the UCI CERTs and GUIDs:

- Message: a declaration of some type of data that is exchanged between components. Each Message is identified using a name that is unique among all Messages. This identifier is known as the Message's Name.

- Message Type: a declaration of the contents of a Message. Each declared Message has an associated message type declaration (a one-to-one relationship).

- uci:MessageType: a declaration of the base message type from which all message type declarations are derived. In the UCI Schema, this is defined with the abstract data type named “uci:MessageType”.

- MessageData: a declaration of the message specific data contained within a Message. Each declared Message has an associated MessageData declaration (a one-to-one relationship).

- Element: a base type defined in the UCI XSD Schema. These are identified with the declaration of xs:element.

- Type: declaration of a data type used by an Element. This includes complexType, simpleType, and embedded defined types.

- Field: an element defined as a child of a sequence

- Choice Item: an element defined as a child of a choice, the collection of which make up the possible selections of that choice.

- UCI CERT: A CERT is a restriction on the UCI Schema that is required and is verifiable using XPath queries or programmatic methods.

- UCI Guideline: A Guideline is a restriction that should be considered during UCI Schema development.

- UCI XSD Schema: Refers to the XSD Schema file UCI_MessageDefinitions_M_N_Px_E.xsd subject to the CERTs and GUIDs documented in this specification.

Follow these steps to create a new message:

- Open the message template with an editor

- Find “MsgName” throughout the template and replace it with the new message name

- Search the schema in text mode for “schemaLocation:” Replace both the UCI_SecurityMarkings_v2_xx.xsd file and UCI_MessageDefinitions_v2_xx.xsd correct names

- Search the schema in text mode for “INSTRUCTIONS:” Replace the annotation text with the new message annotation text

- Search the schema for “AnyRequiredMessageUniqueFieldsGoHere.” Delete that element and replace with the new message required element(s)

- Search the schema for “AnyOptionalMessageUniqueFieldsGoHere.” Delete that element and replace with the new message optional element(s)

- Required and Optional elements may be mixed as appropriate

- The new message is ready to test, or you can copy and paste the new message into an existing schema with other UCI messages to test.

- Do not change any field names that are in the templates, unless specified to do so in the INSTRUCTIONS.

- Only edit annotations that start with “INSTRUCTIONS:”
